# Progress: Python-to-EmojiASM Transpiler

## Original Goal
Build a Python-to-EmojiASM transpiler that compiles a subset of Python (arithmetic, variables, loops, conditionals, random()) to EmojiASM bytecode for GPU execution via the ast module. Agents write Python, we compile to EmojiASM and run 10,000 instances on GPU.

## Status
Spec phase complete. Ready for implementation.

## Learnings
- Synthesis insight from quick mode: The existing `compiler.py` (C AOT compiler) provides an excellent reference pattern for traversing a `Program` and emitting target code. The transpiler inverts this: it *produces* a `Program` from Python instead of consuming one.
- Pattern found: `Instruction(op=Op.X, arg=val, line_num=n, source=src)` is the universal instruction format. The `source` field can hold the original Python line for debugging.
- Pattern found: `disassemble()` in `disasm.py` reconstructs EmojiASM source text from a `Program`. The transpiler can reuse this for `transpile_to_source()` with zero additional code.
- Pattern found: `EmojiASMTool.execute()` already has a `parse -> tier -> route -> execute` pipeline. Adding `execute_python()` only requires replacing the `parse()` step with `transpile()`.
- Division semantics gap: Python `/` always returns float, but EmojiASM `DIV` does floor division for int//int. Solution: coerce left operand to float with `PUSH 1.0, MUL` before `DIV` when compiling `ast.Div`.
- No native `!=`, `<=`, `>=` in EmojiASM: must compose from `CMP_EQ`+`NOT`, `CMP_GT`+`NOT`, `CMP_LT`+`NOT` respectively.
- Label names in `Function.labels` can be plain strings (no emoji required). Labels are internal to the Function and don't appear in user-visible output.
- The VM's memory cells are global across all functions. Function-scoped variables in the transpiler need unique cell names to avoid collision. Using a `VarManager` with scope-prefixed assignment solves this.
- `ast.parse()` is extremely fast (<1ms for typical programs). The entire transpilation pipeline should easily meet the <10ms NFR.
- For `for i in range(n)`, the AST represents it as `For(target=Name('i'), iter=Call(func=Name('range'), args=[Name('n')]))`. Must pattern-match this specific structure.
- Chained comparisons (`a < b < c`) are complex to compile for a stack machine (need to DUP intermediate values). Deferring to V2 and suggesting `a < b and b < c` as workaround is the right V1 tradeoff.
